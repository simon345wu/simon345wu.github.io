## 📝 隨手筆記 / 今日心得
- 回到咖啡粒徑分析這個 side project。之前在 GitHub 找到天文學家 Jonathan Gagné 寫的 coffeegrindsize,跟我的做法完全不同路數:我用 Otsu 二值化 + watershed 分水嶺,他只看**藍色通道**、跟背景中位數比閾值,然後自己手寫 flood fill,黏在一起的粉是靠「沿路徑算亮度成本、遇到亮的山脊就切開」這招分開的。想比較兩者,就得先把架構弄成演算法可以插拔的樣子。
- 於是生出了 `psa` 這個 Python 比較平台。設計上學到最重要的一課是**公平比較**:每個演算法都回傳 label mask,共同指標(等效圓直徑、D10/50/90)用同一套公式從 mask 算 — 不然比出來的是量測公式的差異,不是分割能力的差異。
- 移植 coffeegrindsize 最有意思的部分是驗證。它是一個 3222 行的 Tkinter GUI,演算法跟介面攪在一起,而且檔案結尾沒有 `__main__` 保護,一 import 就會開視窗。Claude 的解法是把原始碼截斷到 class 定義為止、exec 之後用假的 Tk 物件「無頭」驅動原版方法,跟移植版在同一塊裁切上對照 — **166 顆的所有量測值和每一顆的像素清單都逐位元一致**。這種驗證強度讓我後面完全不用擔心移植走樣。
- 對比結果比我預期的有趣:兩個演算法的 **Dn50 每檔差不到 2%**,等於互相印證了正確性。但 Dv50(體積中位數)一開始差了兩倍以上 — 追下去發現是**結塊處理哲學**不同:我有尺寸/圓形度上限過濾,他沒有,過載照片裡的蛇形大結塊被他整團當成一顆,體積統計直接爆掉。加了「共同過濾」(同一把尺套到所有演算法)之後,最誇張的樣本從 5353 µm 掉回 1375,跟我的 1353 幾乎重合。
- 最大的結論其實不是演算法:**拍攝流程比演算法影響更大**。每檔的第一張照片粉撒太多、黏成一片,是所有誤差的最大來源。重拍了一組薄層均勻分散的 3/5/7,兩個演算法的 Dv50 都變成單調遞增、差距縮到 4~10%。以後拍照就是:薄層、分散、不要堆。
- 下午把兩個演算法都搬進 Flutter App。介面設計用了**宣告式參數規格(ParamSpec)**:每個演算法宣告自己有哪些參數(範圍、預設值、要不要收進進階區),控制面板直接照規格自動長出滑桿和開關。以後加第三個演算法,UI 一行都不用改。
- Dart 移植的保真也是同一招:Python 端輸出 fixture,Dart 測試逐值比對。有個細節卡了一下 — numpy 預設的 argsort 是**不穩定排序**,等值的順序跟 Dart 對不起來,最後兩邊都改用穩定排序對齊(tie 的順序本來就是任意的,不影響演算法語意)。端到端在真實照片上,Dart 版和 Python 版的 Dv50 都是 935.60 µm,小數點後兩位相同,很滿意。
- 踩了一個經典的 Dart 坑:分析移到背景 isolate 用 `Isolate.run(closure)`,結果 App 一跑就「分析失敗: isolate」。原因是 closure 會夾帶方法裡其他閉包(setState)共用的環境,把不能跨 isolate 的 State 物件一起塞進去。改成 `compute(頂層函式, 純值參數)` 就好了。教訓:跨 isolate 的東西,明確傳參最穩。
- 另外收了幾個舊帳:專案搬過家,CMake 快取還指向舊路徑 `C:\python_prj`;測試用的合成圖某次重生成之後根本沒有校正框,auto-calibrate 測試其實早就壞了;還有一張過曝照片 ArUco 標記泛白偵測不到,加了自動救援(放寬參數 → 對比正規化重試),Python 跟 Dart 兩邊都有。
- 感想:今天整條線 — 讀別人的演算法 → 抽出來驗證 → 公平比較 → 有結論 → 移植進產品 — 一天走完,中間每一步都有逐值驗證墊底,所以走得很快也很放心。「先建比較平台再做結論」這個順序是對的。
