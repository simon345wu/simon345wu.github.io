## 📝 隨手筆記 / 今日心得
- 今天處理首頁那個「public roast profile」列表。原本是按時間把全部烘焙紀錄一次列出來,越積越多會太長。改成:**Latest 用數字頁碼分頁**、另外加一個 **By Lab 下拉選單**可以只看單一烘焙室的紀錄。
- 需求是邊做邊長出來的,分了三步:先做「只顯示最近 5 個 + View All」→ 我說乾脆「分頁顯示全部」→ 又想到「手機根本做不了 3×3 的排版」,所以再補 RWD。最後結果是桌機每頁 9 張(3 欄)、平板 2 欄、手機 1 欄每頁 6 張,分頁列在手機上只留箭頭。
- 踩到一個有意思的點:為了「顯示全部」,把抓取上限從 50 提高到 500,結果**多跑出一個原本被 50 筆蓋掉的第二個烘焙室(Outlook Coffee)**。原來一直以為只有一家,是被 limit 截掉了沒發現。
- Claude 這次是真的開瀏覽器把不同螢幕尺寸(375 / 768 / 1280)一個個實測給我看,不是只跑 typecheck 就說好了,這點蠻踏實的。
- 學到 / 補齊的:以前開 PR 是走 GitHub API 繞一圈,今天乾脆把 **`gh` CLI 用 winget 裝起來**,自己 `gh auth login` 登入一次,之後 Claude 就能直接 `gh pr create`。開 PR 這件事順很多。
- 一樣走完整套流程:改 → 瀏覽器實測 → 開分支 commit → PR → 我自己按 Merge → 自動部署 → 確認上線。這次確認部署學到:backend 那個 API 版本根本沒回傳 traffic 欄位,與其糾結 rollout 狀態,**直接打開線上網站看新功能有沒有出來最準**。build READY、rollout SUCCEEDED,線上也確認分頁跑出來了。
- 感想:同一個「頁面優化」需求,拆成幾個來回反而更清楚自己要什麼。而且工具鏈每補一塊(像今天的 gh),之後就少一道手續。
